# Agent-Ready Audit

Date: 2026-05-14

This audit tracks the specific goal of making Amend.sh agent-ready across `https://amend.sh` and
`https://docs.amend.sh`.

## Success Criteria

- `https://amend.sh/robots.txt` and `https://docs.amend.sh/robots.txt` use maximum-visibility
  crawler policy.
- Both hosts publish canonical `sitemap.xml` files.
- Both hosts expose LLM-friendly maps or Markdown mirrors through `llms.txt`/`llms-full.txt`.
- Public pages include canonical metadata and social metadata.
- The marketing homepage includes visible, crawlable product copy plus structured data.
- Authenticated/private app routes are excluded from indexing with `noindex,nofollow`.
- Local source/build checks and live production checks prove the behavior.

- `bun run agent-ready:completion-audit` maps the prompt requirements to source artifacts, saved
  report evidence, strict readiness, production env, live validation, and the documented handoff; it
  exits non-zero until the public production report is fully green. `bun run
agent-ready:refresh-report` runs the same audit with `--allow-production-blockers` so source and
  artifact regressions still fail while external DNS/env setup remains incomplete.
  `bun run agent-ready:final-gate` regenerates the saved reports after production env and DNS are
  ready, runs the strict completion audit, refreshes and validates the no-secret status report,
  requires the production, live, status, and completion audit reports to be green, and verifies
  synced audit evidence.
  `scripts/agent-ready-completion-audit.ts --json --json-file` can save the same checklist as
  `agent-ready-completion-audit-report.json`.
- `bun run agent-ready:status` reports the remaining production env and DNS blockers without
  printing secret values; `bun run agent-ready:status:json` emits structured output, and
  `scripts/agent-ready-status.ts --json --json-file` writes the same no-secret preflight status to
  an artifact file with machine-readable `blockers` and `nextGates`.
- `bun run agent-ready:status:validate-report agent-ready-status.json` validates the no-secret
  status artifact against `docs/agent-ready-status-report.schema.json`, including exact next-gate
  commands. Add `--require-ok` when CI should fail unless production env and DNS are both ready.
- `bun --silent run agent-ready:live:json` emits a machine-readable failure report while DNS is still
  blocked. `bun --silent scripts/agent-ready-live.ts --json --json-file agent-ready-live-report.json`
  saves that report, and `bun run agent-ready:live:validate-report agent-ready-live-report.json`
  validates it against `docs/agent-ready-live-report.schema.json`.
- `bun run readiness:strict` is intentionally failing locally until production deployment/provider
  values are loaded; the current strict gate passes `128/149` checks and reports `21` missing
  production env inputs.
- `bun run agent-ready:production` is intentionally failing locally at the strict readiness step
  until those production values are loaded, so it does not continue to built artifact or live public
  checks in an unconfigured environment.
- `bun --silent run agent-ready:production:json` emits one combined CI/deploy JSON report for strict
  readiness, built artifact validation, no-secret status blockers, and the live gate; locally it
  reports failing readiness/live blockers until production values and DNS are loaded. Use
  `scripts/agent-ready-production.ts --json --json-file` to save the combined report as an artifact.
  This keeps strict readiness and built artifact validation visible even when live DNS is still
  blocked. The report includes a `$schema` field,
  `docs/agent-ready-production-report.schema.json` documents the artifact shape, and
  `bun run agent-ready:production:validate-report` validates saved report artifacts. Use
  `bun run agent-ready:sync-audit agent-ready-production-report.json` to sync the latest saved
  status/live timestamps into this audit and the completion audit, then
  `bun run agent-ready:sync-audit:check agent-ready-production-report.json` to verify they stayed
  synced and that `agent-ready-live-report.json` still semantically matches the embedded production
  live report except for `checkedAt`, independent of JSON object key order.
  `bun run agent-ready:audit:check` runs production report validation, status report validation,
  live report validation, completion audit report validation, and audit sync checking together.
  `scripts/agent-ready-production-env.ts` is the source of truth for the 21 production env keys
  used by strict readiness and status.

## Remaining Work

The repo/build side is ready. Public completion requires external domain setup:

1. Follow `docs/agent-ready-domain-setup.md`.
2. Register `amend.sh`.
3. Delegate it to the chosen DNS provider.
4. Deploy the web app for `https://amend.sh`.
5. Deploy the docs app for `https://docs.amend.sh`.
6. Point DNS records at those deployments.
7. Load production env/provider inputs and run `bun run readiness:strict` until it passes.
8. Run `bun run agent-ready:production` until strict readiness, built artifacts, and the live gate
   all pass.
9. Run `bun run agent-ready:completion-audit` until the prompt-to-artifact completion audit passes.
10. Run `bun --silent run agent-ready:production:json` when CI/deploy logs need the combined strict
    readiness, built artifact, status blocker, and live gate report.
11. Run `bun run agent-ready:live` until every check passes when rechecking already-deployed hosts.

Do not call the public site agent-ready until the live validator passes against both production
hosts.
